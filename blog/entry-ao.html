Hackeando AO20

Es noviembre 2022. El creador del Argentum Online vuelve a la escena para lanzar AO20. Para los que no conozcan el AO, el juego es un MMORPG argentino de fin de los 90s de mundo abierto donde hay que grindear mucho. Es de código abierto y tanto servidor como cliente están armados en Visual Basic 6. VB6 se deprecó en abril 2008, pero lo siguen usando para este juego al día de hoy.

Es un juego muy lindo que nos marcó a muchos pibes. El hecho de ser de código abierto también fue una puerta de entrada para muchos al mundo de la programación y de las computadoras en general. Gente que moddeaba y armaba comunidades

Bueno la cuestión es que yo de chico intentaba duplicar items o replicar distintos bugs o hacks para lograr alguna pequeña ventaja sin mucha suerte. Tenía muchas ganas pero pocos resultados. Cuando anunciaron una nueva versión hecha por el creador del juego pensé que sería divertido darla una nueva chance. Para el fin de esta cadena de vulnerabilidades pude spwanear una espada test en Ullathorpe (la ciudad principal) o tomar control de cualquier cuenta.

Lo que sigue a continuación es un writeup de cómo fui vulnerando los sistemas de AO20 para spawnear cualquier item que yo quisiese o tomar control de cualquier cuenta. Lo cierto es que no fue un hack avanzado ni sofisticado, recordemos que este tipo de juegos apenas tiene los recursos para mantener los servidores online y hacer algunos desarrollos por contribuciones (salvo algún que otro servidor que hace algo de plata pero no es la regla).

Pero sí es una dedicación a encontrar descuidos de los demás.

La preparación

Al ser un juego de código abierto uno puede buscar vulnerabilidades en el juego mismo, tanto servidor como cliente. Ese fue mi primer intento. El problema es que están escritos en Visual Basic 6. Me pareció poca atractiva la idea de aprender un stack tan viejo que está completamente fuera de uso en el mundo real, así que después de poco tiempo busqué otros caminos.

Mirando los repos de github de la organización de AO20 me encontré que tenían varios servicios satélites corriendo para distintas funcionalidades de la página web. Uno que captó la atención fue "argentum20-estadisticas".

argentum20-estadisticas

Como su nombre lo indica este repo corre algunos reportes de estadísticas. Está escrito en php. Me llamó la atención reports.php que hacía lo siguiente:

<?php
  include('_reports.php');
  $directory = '../' . $_GET['dir'];

  if (isset($_GET['last']) && $_GET['last'] == 'true') {
    echo getLastReport($directory);
    die();
  }
  
  foreach (listFolderFiles($directory) as $file) {
    echo '<a href="' . $file .'">'. $file .'</a><br>';
  }
?>

spot the vuln!

El parámetro dir se concatena directo a la ruta sin sanitizar. Es un simple Path Traversal (LFI).

El parámetro last=true descargaba el último archivo modificado. Sin este parámetro, el script listaba directorios. Se usó para eludir restricciones y acceder a carpetas internas.

Entonces puedo hacer, por ejemplo: GET https://estadisticas.ao20.com.ar/produccion/reports.php?dir=../backups, que esto me liste los files que hay en backups. Y despues puedo leer el contenido de cualquiera de estos files agregando el queryparam last en true.

Con este LFI en mano me puse a revisar todas las carpetas y files que había en este servidor... y así fue como encontré...

Credenciales FTP expuestas

Revisando directorios, llegué a la carpeta de backups. El servidor de estadísticas tenía permisos de lectura sobre carpetas que no debería. Ahí encontré logs de Jenkins y archivos de configuración que tenían credenciales FTP en texto plano:

ftp://estadisticas@ao20.com.ar:[REDACTED]@[REDACTED]

O sea que ya no necesitaba el LFI para listar carpetas; tenía credenciales ftp. Y además esto me daba permisos de escritura. A esta altura ya podía subir un archivo malicioso php y tomar control del servidor, pero no me interesaba ese camino en particular.

Backups y Account Takeover

Descubrí que el servidor del juego (SQLite) se backupeaba cada 4 horas en ../backup/server-prod-bd. Usando el LFI, descargaba el archivo y tenía la base de datos completa de los jugadores en mi PC.

Acá me quedé pensando qué podía hacer con un backup de la base de datos... y tuve una idea.

Qué tal si antes de que se escriba el backup yo pido recuperar la contraseña de una cuenta? El token de cambio de clave se tiene que guardar en la db... y yo podría después descargar este backup y usar ese token para cambiar la contraseña de cualquier cuenta. Y eso fue lo que hice. Busqué alguna cuenta de nivel alto que no se usara desde hacía mucho tiempo y pedí prestada su cuenta.

Así que el flujo era:

1. Pedir un "Olvidé mi contraseña" en la web para la víctima.
2. Esperar al siguiente cronjob de backup.
3. Bajar la DB, buscar el token de recuperación recién generado.
4. Usar ese token en la web y cambiar la clave.

Con esto ya podía pedir prestadas cuentas y divertirme un rato, pero... nada más??

Jenkins

Creo que estuve trabado en este paso por un par de noches. El ftp estaba repleto de archivos y carpetas. Así que estuve unas cuantas horas revisando todo en búsqueda de algo que me permita seguir avanzando.

Siguiendo con la exploración de backups, encontré un .zip viejo de una instalación de Jenkins. Adentro estaba credentials.xml con las llaves SSH de GitHub, pero encriptadas.

Sin embargo, el mismo .zip tenía la master key del Jenkins. Con la llave maestra y los hashes, pude desencriptar la clave privada de SSH. Con eso cloné todos los repositorios privados de AO20: pymmo, gameapi y el engine. Ahora tenía el mapa completo de cómo funcionaba todo por detrás.

El fin

En el código fuente de gameapi encontré lo que buscaba: las credenciales de la base de datos MySQL de producción (ao20_mercado).

El sistema de mercado funcionaba con un job que leía una tabla de transacciones. Si insertabas un registro ahí, el juego te daba el item. Armé un script rápido para inyectar la compra:

mysql -u root -p[REDACTED] -h [REDACTED] -P 5103 ao20_mercado \
  -e "insert into mao_item_transaction_status \
  (user_id, status, created_at, item_id, item_quantity) \
  values ('8304', 'CONFIRMED', NOW(), '1745', '1337');"

Elegí el ID de la Espada Test (un item que obviamente no se consigue dentro del juego). Un minuto después, el item apareció en mi inventario. Se ejecutaba un DELETE sobre el registro instantes después de que el job del mercado lo procesara para no dejar rastros en el panel.
